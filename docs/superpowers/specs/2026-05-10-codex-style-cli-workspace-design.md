# Codex 风格 CLI 工作台设计

## 概要

本设计为 AgenticBoot 增加一个面向小白的统一工作台，用来承接“安装完成后立刻可用”的使用场景。

工作台的目标不是重做各家桌面端，也不是实现一个完整 IDE，而是提供一个 **Codex 风格的单会话界面**：

- 前台体验对齐 Codex 桌面版的会话壳
- 底层继续通过各家 CLI 执行任务
- 桌面版工具继续只负责安装、检测和启动入口，不进入统一会话执行层

这项设计解决的问题是：当前 AgenticBoot 已经较好覆盖“装”和“检”，但对小白来说，“装完以后怎么开始用”仍然存在明显摩擦。

## 目标

- 让用户安装完成后不必再接触 PowerShell、VS Code、Cursor 等外部入口，也能直接开始使用 AI 编程代理。
- 提供一个统一、轻量、低学习成本的工作台，而不是一组工具卡片加启动按钮。
- 在视觉和主流程上尽量贴近 Codex 桌面版的使用习惯。
- 统一承接多种 CLI 工具，但前台不暴露工具之间的交互差异。
- 支持最小但完整的任务闭环：新建会话、选择项目、发送任务、查看结果、查看变更、预览并轻编辑文件。

## 非目标

- 不重做 Codex、Claude Code、Gemini CLI、OpenCode、OpenClaw、Hermes 的底层代理内核。
- 不把现有桌面版工具接入统一会话执行层。
- 不在一期实现完整 IDE 能力，例如文件树、多标签编辑、代码补全、复杂 diff 编辑器、调试器等。
- 不在一期建设插件生态、自动化体系、多用户协作或完整 Git 客户端。
- 不要求各家 CLI 在一期就支持完全一致的细粒度事件协议。

## 产品定位

这不是“内置 chat 页面”，而是一个 **统一 CLI 工作台**。

对用户来说，它应该看起来像一个会做事的助手：

- 左边看到项目和历史会话
- 中间直接说需求
- 右边查看改了什么、做了什么、打开文件看看
- 底部只保留少量关键上下文，例如当前工具和当前目录

对系统来说，它是一个 **Codex 风格前台 + CLI 适配后端**：

- UI 和信息架构向 Codex 靠拢
- 执行仍然由现有 CLI 完成
- AgenticBoot 负责把 CLI 行为重组为用户能理解的会话过程

## 一期范围

### 必须有

- 新的工作台主视图
- 左栏项目与会话列表
- 中栏单会话主视图
- 右栏“变更”和“文件”两个区域
- 底部统一输入区
- 工具下拉切换
- 工作目录选择
- 会话保存与恢复
- 本轮任务的文件改动列表
- 右栏单文件预览
- 右栏单文件轻编辑与保存

### 可以晚一点做

- 更细的来源展示
- 更丰富的 Git 操作
- 更复杂的权限控制 UI
- 更完善的命令日志结构化
- 多标签文件编辑
- 文件树浏览器

## 页面结构

### 左栏：项目与会话

左栏负责提供上下文，而不是工具堆叠。

建议包含：

- `新对话`
- `搜索`
- 按项目分组的会话列表

项目下展示最近会话标题即可，不做复杂批量操作，也不带管理器式的大量筛选器。一期重点是“找到上次在这个项目里做过什么”。

### 中栏：单会话主视图

中栏是整个产品的主舞台，默认只强调会话和任务过程，不默认摊开原始终端输出。

中栏展示的不是“CLI 原始回显”，而是 **任务时间线**。每轮任务至少有这些阶段：

- 用户输入
- 代理理解与计划
- 执行中状态
- 等待确认
- 结果总结

原始日志、详细命令、复杂报错不应挤占主视图，而应作为次级信息进入右栏或折叠细节。

### 右栏：变更与文件

右栏保持轻量，只做两个一级区域：

- `变更`
- `文件`

#### 变更

展示本轮任务涉及的文件结果：

- 新增
- 修改
- 删除

默认先显示文件列表和简短摘要，不要求一期就做完整高级 diff 工具。

#### 文件

支持选中单个文件后：

- 预览文本内容
- 轻量编辑
- 保存回磁盘

一期只支持单文件打开和编辑，不做多标签、复杂代码智能能力或完整工作区浏览。

### 底部输入区

底部输入区保持“像聊天”，但它本质是任务输入区而不是纯聊天输入区。

建议包含：

- 主输入框
- 当前工具下拉
- 当前工作目录状态
- 可选的运行模式或权限状态

不建议在底部塞入大量按钮或复杂工具条，避免前台退化成 IDE。

## 交互原则

### 像 Codex，而不是像终端

用户不应该面对一整屏原始 CLI 输出。工作台需要把底层执行过程翻译成用户能理解的结构化过程。

例如：

- “正在分析项目”
- “正在执行命令”
- “需要你确认是否继续”
- “已修改 3 个文件”

而不是直接把 ANSI 输出、spinner、原始交互提示裸露给用户。

### 像工作台，而不是像 IDE

一期不追求把所有开发操作都搬进来，只解决最小闭环：

- 发起任务
- 看它执行
- 看它改了什么
- 需要时手动改一个文件
- 继续下一轮

### 像面向小白的产品，而不是面向熟练终端用户的壳

默认不要求用户先懂：

- 哪个 CLI 最适合
- 怎么开 PowerShell
- 怎么切工作目录
- 怎么看日志判断是否成功

这些复杂度应被产品吸收。

## 会话模型

一期建议在 AgenticBoot 内部定义一个最小统一事件模型，而不是要求所有 CLI 一开始就有完美一致的协议。

推荐最小事件类型：

- `assistant_text`
- `status`
- `command`
- `file_changed`
- `approval_required`
- `completed`
- `error`

这套模型的目的不是精准表达每家 CLI 的全部内部状态，而是为前台提供一个稳定的数据骨架。

如果某个 CLI 暂时无法输出细粒度事件，也允许退化为：

- `status`
- `assistant_text`
- `completed`
- `error`

前台仍然能够工作，只是细节较少。

## 工具策略

一期前台是强统一的：

- 同一个主界面
- 同一个输入框
- 同一个任务时间线
- 同一个右栏结构

底层则通过适配层驱动各家 CLI。

推荐纳入统一工作台的对象：

- Codex CLI
- Claude Code CLI
- Gemini CLI
- OpenCode CLI
- OpenClaw
- Hermes

桌面版工具继续留在安装与启动层：

- Claude Code Desktop
- Codex Desktop
- OpenCode Desktop

## 最小可行版本

### MVP 必须完成的闭环

1. 用户从 AgenticBoot 进入工作台。
2. 用户新建一个会话。
3. 用户选择或绑定项目目录。
4. 用户选择一个 CLI 工具。
5. 用户发送任务。
6. 工作台显示执行中的状态和文本结果。
7. 如果发生文件改动，右栏展示本轮变更列表。
8. 用户可以点开某个文件预览。
9. 用户可以直接轻量编辑并保存该文件。
10. 用户继续下一轮任务。

### MVP 暂不要求

- 完整 diff 浏览器
- 多文件并行编辑
- 文件树导航
- 完整 Git 工作流
- 搜索全项目内容
- 富文本结果物系统
- 完整工具能力一致性

## 与当前 AgenticBoot 的关系

这项设计属于 AgenticBoot 的新增主线能力，不是对现有安装器的小修小补。

但它仍然符合当前仓库的二开方向，因为它直接延伸了“安装、检测、工具管理”之后最自然的一步：**让工具在装完以后直接可用**。

它不要求先深挖上游 CC Switch 全量模块，而应优先依托当前二开主战场周边的前端壳子和会话相关能力增量实现。

## 实现切分

### 前端

新增一个独立的工作台主视图，不从 `Wizard` 或 `Manager` 直接演化。

建议新增区域：

- 工作台主页面
- 左栏会话导航
- 中栏消息与任务时间线
- 右栏变更/文件面板
- 底部输入区

适合优先复用的现有基础：

- `src/components/sessions/`
- `src/components/ui/`
- 安装会话和事件状态相关 hooks 的事件思路

### Tauri / Rust

一期后端只补最小闭环能力：

- 启动 CLI 会话
- 对已有会话继续输入
- 返回或推送统一事件
- 处理确认继续
- 读取文件
- 写入文件
- 提供本轮改动文件列表

不要求一期就建设完整跨工具富协议。

## 风险与取舍

### 风险 1：退化成“终端换皮”

如果前台只是显示原始 CLI 输出，用户仍会觉得在看命令行，产品价值会被削弱。

缓解方式：

- 中栏只展示任务过程
- 原始日志弱化
- 右栏只展示高价值结果物

### 风险 2：范围膨胀成“再做一个 IDE”

如果一期把文件树、多标签、Git、diff、终端、插件都塞进来，会大幅拖慢交付。

缓解方式：

- 严格限制右栏只做“变更”和“文件”
- 文件只做单文件预览与轻编辑
- 不引入复杂工程化编辑器能力

### 风险 3：不同 CLI 行为差异导致统一困难

虽然前台统一，但不同工具的交互模型并不完全一致。

缓解方式：

- 先定义最小统一事件模型
- 允许个别工具一期先提供较少事件
- 保证主流程统一，而不是追求内部语义完全一致

## 设计结论

AgenticBoot 有必要内置统一工作台，但正确方向不是“做一个内置聊天页”，而是：

**做一个 Codex 风格的统一 CLI 工作台。**

一期的最合适边界是：

- 前台模仿 Codex 的轻量会话壳
- 底层继续调用各家 CLI
- 只纳入 CLI 执行层，不纳入桌面端执行层
- 右栏支持变更查看与单文件预览/轻编辑

这个范围既能明显降低小白摩擦，也不会把项目过早拖入重做 IDE 或重做 agent runtime 的复杂度中。
